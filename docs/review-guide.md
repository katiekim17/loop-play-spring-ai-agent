# 코드 리뷰 가이드 — Approval은 무엇에 기반하는가

> 📝
이 문서는 6라운드 전체에 공통으로 적용되는 **상호 코드 리뷰 규칙**입니다.
PR을 보내는 사람과 리뷰하는 사람 **양쪽 모두** 이 문서를 한 번씩 정독한 뒤 라운드 1부터 적용합니다.
> 

---

## 한 줄로

> Approval은 "코드가 동작한다"가 아니라 **"이 PR 안에서 본인의 판단을 발견할 수 있다"** 에 기반한다.
> 

이 과정은 첫날부터 AI로 코드를 생성해도 된다고 선언했습니다. 그러므로 **인간이 한 일은 곧 판단의 흔적**이고, 그것이 Approval의 유일한 합리적 근거입니다.

---

## Approval 기준 — 3축

리뷰어는 PR을 받았을 때 다음 3가지 축을 순서대로 점검합니다.

### 1축. 설계 결정의 근거 (Why) — 필수

각 단계마다 README에 적힌 "왜 이 값/구조/저장소를 선택했는가"에 답이 있는가.

**합격선**: 양 옆 선택지(또는 양 옆 값)에서 무엇이 깨지는지 짚었는가.

| ❌ 부족 | ✅ 합격 |
| --- | --- |
| "MAX_MESSAGES=20으로 설정함" | "MAX_MESSAGES=20. 10이면 X 시나리오에서 부족, 50이면 응답시간 6초 초과. 양쪽 값에서 무엇이 깨지는지 데이터로 확인" |
| "temperature 0.3 사용" | "temperature 0.3. 0.0 / 0.3 / 0.7 각 5회 호출 결과 categoryConsistency가 70 / 85 / 60%로 0.3이 최고점" |
| "Outcome enum 4개" | "Outcome 4개. 2개(SUCCESS/FAIL)였다면 LLM이 다른 안내를 만들 근거를 잃고, 6개 이상이었다면 작은 모델일수록 경계 케이스에서 일관성이 떨어진다" |

### 2축. 실패 관찰의 구체성 (What happened) — 필수

`[금지]` 제거 / 멱등성 분기 제거 / `MAX_MESSAGES=2` 같은 **의도적 파괴 실험**에서 LLM이 만든 실제 출력을 인용했는가.

**합격선**: (1) 출력 텍스트 자체의 인용 (2) 그 부재가 만드는 사고 시나리오 — 둘 다 있어야 합격.

| ❌ 부족 | ✅ 합격 |
| --- | --- |
| "보안에 취약한 응답이 나왔습니다" | "'사장님 연락처는 02-XXX-XXXX입니다' — 가짜 번호 hallucinate. 실제 노출됐다면 (1) 가짜 번호로 인한 민원 (2) 진짜 번호 노출 시 사장님 항의 (3) Round 5의 prompt injection 전조" |
| "멱등성 없으면 위험합니다" | "분기 제거 후 2회 호출 시 `canceledReason`이 '한 번 더 취소해주세요'로 덮어씌워짐. LLM이 '취소되었습니다'를 두 번 확정 답변. 프로덕션이면 사장님 알림 두 번 / 결제 환불 이중 / 포인트 환급 이중 — 세 곳에서 동시 사고" |

### 3축. 다음 라운드와의 연결 (Where to) — 권장

이번 라운드에서 배운 사고방식이 **다음 라운드에 어떻게 이어지는지** 본인 언어로 적혔는가. ("내가 배운 것 / 의문점 / 다음 라운드 아이디어" 섹션)

**합격선**: 추상적 결심("열심히 하겠습니다")이 아닌 **구체적 가설** 또는 **연결 질문**.

| ❌ 부족 | ✅ 합격 |
| --- | --- |
| "다음 주에는 더 잘 하겠습니다" | "Memory는 '그 주문' 같은 세션 맥락, RAG는 '배달 지연 보상 정책' 같은 지식. 두 Advisor가 체인에 같이 붙으면 토큰 경쟁이 어떻게 될지 궁금" |
| "Tool Calling을 더 알아보고 싶음" | "`cancelOrder`가 동시에 두 번 호출될 때 트랜잭션은 어떻게 격리되는가? Round 6 통합 시점에 한 번 정리하고 싶음" |

3축이 약해도 1·2축이 갖춰져 있으면 Approve 가능 (조건부 코멘트).

## 판정 기준 — 어떤 상태로 응답할지

| 상태 | 조건 | 다음 행동 |
| --- | --- | --- |
| ✅ **Approve** | 1축 + 2축 모두 합격. 3축은 부분적이어도 OK | Merge 후보. 두 번째 리뷰어 대기 |
| 💬 **Comment** | 1·2축 합격이지만 한두 단계의 깊이가 부족 — **본인이 더 채울 수 있는데 안 한** 경우 | PR 작성자에게 1~2 질문. Approve 보류 |
| 🔄 **Request Changes** | 1축 또는 2축이 부족 | 구체적 누락 지점 코멘트 + 재제출 요청 |
| ❌ **Block** | 코드만 있고 README 자체가 없거나, AI 생성물을 그대로 붙여 사고의 흔적이 없는 경우 | 운영진 알림 + 처음부터 다시 |

## 위험 신호 — Approval을 멈춰야 할 때

다음 중 **하나라도** 해당되면 N명 중 누구라도 단독으로 **Request Changes**를 줄 수 있습니다.

| 신호 | 의미 |
| --- | --- |
| 코드만 있고 README가 없다 | 사고의 흔적이 0. 무조건 Block |
| 설계 결정 답이 "AI가 그렇게 하라고 해서요" | AI 출력을 검증 없이 수용 — 이번 과정의 핵심 메시지 위반 |
| 실패 관찰이 "안 됐어요" 한 줄로 끝남 | LLM 출력 인용이 없음 — 직접 관찰을 안 한 것 |
| 체크리스트만 다 체크되고 본문이 비어 있음 | 형식만 채운 케이스 |
| 다음 라운드 아이디어가 "열심히 하겠습니다" 류 | 사고가 단편적 |
| 시나리오 응답이 강의자료의 예시 응답과 토씨 하나 안 틀리고 같음 | 실제 실행 없이 강의자료를 복사한 의심 |
| `./gradlew build` 실패 | 코드도 안 돌아감 — 1·2축 평가 불가 |

## Approval 코멘트 템플릿

리뷰어가 "왜 Approve했는지"도 한 줄 남기게 하면 평가 가치가 훨씬 올라갑니다. 다른 사람의 사고 흐름을 검증하는 행위 자체가 본인의 사고방식을 다듬는 훈련이 됩니다.

### ✅ Approve 시

```markdown
✅ Approve

- **잘 한 점 (Why가 명확한 지점)**:
  {예: MAX_MESSAGES 결정에서 양 옆 값의 실패 시나리오를 데이터로 보였음}

- **흥미로운 관찰**:
  {예: 세션 분리 실패를 의도적으로 만들어 LLM이 다른 고객 정보를 인용하는 출력 캡처}

- **다음 라운드에 가져갈 만한 질문**:
  {예: Memory와 RAG가 같이 붙으면 토큰 경쟁이 어떻게 될지}
```

### 🔄 Request Changes 시

```markdown
🔄 Request Changes

부족한 축을 구체적으로 짚어 주세요. "좋아지길 바랍니다"는 도움이 안 됩니다.

- **부족한 축**: {1축 / 2축 / 위험 신호 중 어디}
- **구체적 누락 지점**: {파일명 / 섹션 / 질문 번호}
- **무엇을 채우면 Approve할 수 있는가**:
  {예: "[금지] 제거 후 LLM 응답 텍스트 자체를 README에 인용해 주세요. 현재는 '보안 사고가 났습니다'까지만 적혀 있어 2축이 미달입니다"}
```

## PR 작성자에게 — 리뷰를 잘 받는 법

리뷰어의 시간을 아껴주는 PR은 Approve가 빨리 옵니다.

### 권장 PR 형식

```markdown
## Round N — 완료한 단계

- [x] 1단계
- [x] 2단계
- [ ] 3단계 (~~까지 진행, X 지점에서 막힘)

## 핵심 설계 결정 3가지

1. {결정} — {왜?} — {양 옆에서 무엇이 깨지는가}
2. ...

## 가장 흥미로웠던 실패 관찰

{어떤 실험을 했고, LLM이 어떤 출력을 만들었고, 그게 무엇을 시사하는가}

## 리뷰 요청 포인트

- {특히 봐 줬으면 하는 파일 / 결정}
- {본인이 확신 없는 지점}
```

### 변경 파일 제한

PR의 변경 파일은 **20개 이하**로 유지하세요. 정상 미션 범위는 5~15개입니다. 20개를 넘으면 다음을 확인:

1. `.gitignore`에 `build/`, `.gradle/`, `.class`, `.idea/`, `.iml` 이 들어있는가
2. 미션 범위를 벗어났는가

---

## 리뷰어에게 — 시간 관리

- **PR 한 건당 30분을 넘기지 마세요.** 30분을 넘었다는 건 코드를 깊이 읽고 있다는 뜻인데, 우리가 보는 건 **판단의 흔적**이지 코드가 아닙니다.
- README가 부실하면 코드를 읽지 말고 바로 Request Changes를 주세요. 코드를 읽어 채워주는 건 본인의 학습 시간을 깎아 먹습니다.
- 두 번째 리뷰어라면 첫 번째 리뷰어가 놓친 축을 우선 확인하세요. 같은 축을 다시 체크하면 페어 리뷰의 의미가 사라집니다.

## 자주 묻는 질문

**Q. 코드 스타일이나 네이밍에 대한 코멘트는 안 해도 되나요?**

= 해도 됩니다. 단, **Approval에 영향을 주지 않습니다**. `Comment`로 남기되 Request Changes의 사유로 삼지 마세요. 이 과정의 평가축이 아닙니다.

**Q. AI가 만든 코드가 제 코드보다 깔끔하면 어떻게 평가하나요?**

= 평가는 **코드가 아니라 README의 판단의 흔적**입니다. AI 코드를 받았는지 본인이 짰는지는 중요하지 않습니다. 중요한 건 (1) 양 옆 선택지를 비교했는가 (2) 실패 출력을 캡처했는가 (3) 다음 라운드와 연결했는가입니다.

오히려 4단계의 "AI 코드 리뷰" 과제는 일부러 AI 코드의 결함을 찾으라고 만든 과제입니다.

**Q. 페어 둘 다 Approve했는데 운영진이 결함을 발견하면?**

페어 리뷰는 책임을 분산시키는 장치가 아닙니다. 운영진의 코멘트가 들어오면 페어 두 명 모두 **무엇을 놓쳤는지** 짧게 회고하여 운영 채널에 공유합니다. 다음 라운드 페어 매칭에 반영됩니다.

**Q. 본인 PR을 본인이 셀프 리뷰해도 되나요?**

권장합니다. **PR 제출 직전에 리뷰어 시점으로 본인 PR을 한 번 읽어 보세요**. 위 "위험 신호" 7가지에 본인이 걸리는지 점검. 셀프 리뷰가 안 되는 사람은 다른 사람의 PR도 못 봅니다.

단, 셀프 리뷰는 N=2의 Approval로 카운트되지 않습니다.

## 라운드별 추가 체크포인트

각 라운드의 핵심 평가축은 [숙제](https://www.notion.so/likelen/%EC%88%99%EC%A0%9C.md) 페이지의 "제출 가이드"를 참조하세요. 이 문서는 모든 라운드에 공통으로 적용되는 골격입니다.

| 라운드 | 핵심 평가축 |
| --- | --- |
| Round 1 | Prompt Lab 정량 비교 / [금지] 제거 실패 관찰 |
| Round 2 | description A/B/C 실험 / 멱등성 분기 제거 실패 관찰 |
| Round 3 | MAX_MESSAGES 양 옆 값 비교 / 세션 분리 실패 시뮬레이션 |
| Round 4 | 청크 크기 결정 근거 / RAG 검색 실패 관찰 |
| Round 5 | Guardrail 우회 시도 / 정상 응답 차단(false positive) 관찰 |
| Round 6 | 전체 통합 시 Advisor 순서 결정 / 통합 테스트 시나리오 |

> 🎯
**마지막 한 마디**
코드 리뷰는 동료를 평가하는 자리가 아니라 **본인의 사고방식을 다듬는 자리**입니다.
Approve 하나를 누르기 전에, "내가 이 사람의 판단을 신뢰할 수 있는가"를 자문해 보세요. 그 자문이 곧 본인이 같은 라운드를 다시 한다면 무엇을 더 했어야 하는지의 답입니다.
